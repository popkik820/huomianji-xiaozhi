# 小智服务器工作流程与项目作用总结

## 1. 服务器在项目中的定位

在本项目中，小智服务器承担的是智能服务中枢的角色。它不是单纯的网页后台，也不是直接控制单个硬件外设的底层程序，而是连接终端硬件、语音交互、大模型推理和业务功能的核心软件节点。

硬件终端负责采集用户输入和现场信息，例如语音、按键事件、设备状态、传感器数据或视觉信息；服务器负责理解这些输入，并结合系统设定、用户画像、上下文记录和大模型能力，生成可执行、可反馈的结果。

可以将整体系统理解为三层：

| 层级 | 主要组成 | 作用 |
| --- | --- | --- |
| 硬件交互层 | ESP32/SC171v2、麦克风、扬声器、屏幕、按键、传感器、执行机构 | 采集输入、播放反馈、执行动作 |
| 终端通信层 | 设备固件、WebSocket 客户端、OTA 配置获取模块 | 连接服务器，上传语音、状态和控制请求 |
| 智能服务层 | 小智服务器、ASR、LLM、TTS、业务逻辑、知识库/营养推荐模块 | 理解用户意图，生成回复和控制建议 |

因此，小智服务器的核心价值在于：

```text
让硬件设备具备自然语言理解、智能决策和可持续扩展的服务能力。
```

## 2. 服务器的核心功能

### 2.1 统一承接设备连接

设备启动后，会先通过 OTA 接口获取服务器下发的配置信息，其中最关键的是 WebSocket 地址。随后，设备通过 WebSocket 与服务器建立长连接。

典型接口包括：

```text
OTA 接口：
http://服务器地址:8003/xiaozhi/ota/

WebSocket 接口：
ws://服务器地址:8000/xiaozhi/v1/
```

OTA 接口负责告诉设备“应该连接哪里”，WebSocket 接口负责后续的实时语音、文本和状态交互。

这种设计使设备端不需要把所有配置写死在固件中。后续如果服务器地址、模型配置或功能参数发生变化，可以通过服务端集中调整，减少反复烧录固件的成本。

### 2.2 管理语音交互链路

小智服务器在语音交互中承担完整的中枢调度作用。设备端采集用户语音后，通过 WebSocket 将音频上传给服务器。服务器再调用语音识别模块，将音频转换为文本，并交给大模型或业务逻辑进行处理。

完整流程如下：

```text
用户说话
  ↓
终端麦克风采集音频
  ↓
设备通过 WebSocket 上传音频
  ↓
服务器进行 VAD / ASR 语音识别
  ↓
服务器调用大模型或业务模块理解意图
  ↓
服务器生成回复文本或控制建议
  ↓
服务器调用 TTS 合成语音
  ↓
设备接收音频并通过扬声器播放
```

在这个过程中，终端主要负责“听”和“说”，服务器负责“理解”和“组织回复”。

### 2.3 调用大模型完成智能理解

服务器可以接入 DeepSeek、ChatGLM 或其他大语言模型，用于理解用户自然语言输入，并生成符合项目场景的回复。

与普通聊天机器人不同，本项目中的大模型并不是开放式闲聊角色，而是被限制在具体任务场景中。例如：

- 识别用户表达的需求；
- 分析用户输入中的关键信息；
- 根据系统提示词保持固定角色；
- 调用工具函数完成特定任务；
- 将复杂结果转化为适合语音播报的自然语言。

通过服务端配置提示词，可以让服务器表现为某一类专业助手。例如在饮食健康场景中，它可以作为家庭营养管家；在视障辅助场景中，它可以作为环境感知与语音交互助手。

### 2.4 支持业务功能扩展

小智服务器不仅能完成语音对话，还可以扩展业务模块。我们此前设计的营养推荐模块就是一个典型例子。

在该功能中，用户可以通过语音或小程序告诉系统自己今天吃了什么、近期饮食情况如何、接下来想吃什么面食。服务器解析这些信息后，可以输出：

- 今日饮食记录；
- 主食和杂粮摄入情况；
- 推荐面团总重量；
- 推荐杂粮比例或杂粮重量；
- 适合和面机执行的参数建议。

例如，用户说：

```text
今天早上吃了全麦面包，中午吃了一碗面条，晚上想吃饺子。
```

服务器可以结合用户画像、食物词库、营养规则和模型解析结果，生成类似：

```json
{
  "target_food": "饺子",
  "dough_total_weight_g": 500,
  "coarse_grain_ratio": 0.30,
  "coarse_grain_weight_g": 114,
  "reason": "今日已有一定精制主食摄入，建议适度提高本次面团中的杂粮比例。"
}
```

这说明服务器可以从单纯“回答问题”，进一步扩展为“理解需求并生成设备可执行参数”的智能决策中心。

## 3. 服务器工作流程

### 3.1 设备接入流程

设备启动后的第一步是接入服务器。流程如下：

```text
设备启动
  ↓
连接 WiFi / 网络
  ↓
访问 OTA 接口
  ↓
获取 WebSocket 地址
  ↓
建立 WebSocket 长连接
  ↓
发送 hello 消息
  ↓
服务器返回 session_id
  ↓
设备进入待唤醒或待交互状态
```

其中，`session_id` 用于标识当前会话。设备后续发送 `listen`、音频或其他消息时，需要携带当前会话信息，保证服务器能正确关联上下文。

### 3.2 唤醒与监听流程

当设备检测到唤醒词后，会通过 WebSocket 通知服务器：

```json
{
  "type": "listen",
  "state": "detect",
  "text": "你好小智",
  "session_id": "当前会话ID"
}
```

服务器收到后，会返回唤醒提示语，例如：

```text
我认真听着呢，请讲。
```

之后设备进入录音状态，并发送：

```json
{
  "type": "listen",
  "state": "start",
  "mode": "auto",
  "session_id": "当前会话ID"
}
```

用户说完后，设备发送：

```json
{
  "type": "listen",
  "state": "stop",
  "mode": "auto",
  "session_id": "当前会话ID"
}
```

服务器据此判断语音输入的开始和结束，并触发后续识别和回复流程。

### 3.3 服务端处理流程

服务器收到用户输入后，会按模块化流程处理：

```text
WebSocket 接收消息
  ↓
判断消息类型：hello / listen / audio / mcp / text
  ↓
如果是音频，进入 VAD 和 ASR
  ↓
如果是文本，进入意图识别和大模型处理
  ↓
根据结果调用工具函数或业务模块
  ↓
生成回复文本
  ↓
调用 TTS 生成语音
  ↓
通过 WebSocket 返回给设备
```

这种流程将连接管理、语音处理、模型推理和业务逻辑拆分开来，便于后续维护和扩展。

## 4. 服务器的技术特点

### 4.1 通信方式实时性强

服务器与设备之间主要通过 WebSocket 通信。相比普通 HTTP 请求，WebSocket 更适合语音交互场景，因为它可以保持长连接，支持服务器主动下发消息和音频流。

这对于语音助手类设备非常重要：

- 唤醒后可以立即返回提示语；
- 音频可以分帧上传和下发；
- 服务器可以实时控制设备播放、停止或切换状态；
- 设备断线后可以自动重连。

### 4.2 AI 能力与硬件控制解耦

服务器将大模型、语音识别、语音合成和业务逻辑放在服务端处理，终端硬件只需要负责采集、播放和执行。

这样有几个好处：

1. 更换模型时不需要修改硬件程序；
2. 调整提示词时不需要重新烧录固件；
3. 增加新功能时优先扩展服务端模块；
4. 终端资源压力更小；
5. 多个终端可以复用同一套服务能力。

### 4.3 配置集中管理

服务器通过 `.config.yaml` 管理运行配置，包括：

- WebSocket 地址；
- OTA 地址；
- 模型供应商；
- ASR / TTS 模块；
- 系统提示词；
- 工具函数列表；
- 角色设定；
- 业务模块参数。

这种集中配置方式让系统调试更加直接。例如，如果需要让设备从局域网运行切换到公网运行，只需要将 WebSocket 地址改成公网可访问地址：

```yaml
server:
  websocket: ws://公网IP:8000/xiaozhi/v1/
```

如果部署在云服务器上，则设备可以跨网络直接访问服务器，不再受同一局域网限制。

### 4.4 支持云端部署和远程接入

服务器既可以在本地电脑上运行，也可以部署到阿里云、百度云等云服务器上。

本地运行适合调试，云端部署适合比赛演示和多人协作。云端部署后，设备只需要访问公网接口：

```text
OTA:
http://云服务器公网IP:8003/xiaozhi/ota/

WebSocket:
ws://云服务器公网IP:8000/xiaozhi/v1/
```

这样，ESP32 或其他终端设备即使不在同一个局域网，也可以连接服务器。

### 4.5 具备业务工具调用能力

服务器可以通过 function call 或插件机制调用外部工具和业务模块。例如：

- 查询天气；
- 获取新闻；
- 播放音乐；
- 调用营养推荐服务；
- 调用视觉分析服务；
- 后续接入设备控制接口。

这使服务器不仅是“对话生成器”，还可以成为“任务执行调度器”。

## 5. 与项目业务的结合方式

### 5.1 在智能和面机场景中的作用

对于全自动和面机项目，服务器可以承担家庭营养管家的角色。用户通过小程序、语音或触控屏输入饮食情况和用餐需求，服务器根据健康饮食规则和用户画像进行分析，生成适合设备执行的和面参数。

业务流程可以概括为：

```text
用户说明今日饮食和想吃的面食
  ↓
服务器解析食物种类、食量、用餐时间和目标面食
  ↓
结合用户画像判断推荐食量
  ↓
计算面团总重量
  ↓
计算杂粮比例或杂粮重量
  ↓
返回给小程序、语音助手或和面机控制系统
  ↓
设备按参数执行和面
```

服务器输出的不只是自然语言建议，还可以输出结构化 JSON 数据，方便设备或小程序直接使用。

### 5.2 在多模态辅助终端中的作用

对于视障辅助或智能交互终端，服务器可以承接语音输入、视觉分析、按键事件和上下文管理。

设备侧负责采集环境，服务器侧负责综合分析。例如：

- 用户通过语音询问周围环境；
- 终端上传图像或视觉分析请求；
- 服务器结合视觉结果和大模型生成描述；
- 终端通过扬声器播报结果。

在此类场景中，服务器的价值是把多源输入统一转化为自然语言反馈，提高用户获取信息的效率。

## 6. 项目优势总结

小智服务器为项目带来的优势主要体现在以下几个方面：

1. **智能化能力集中**  
   服务器统一承载 ASR、LLM、TTS 和业务逻辑，让设备具备自然语言交互能力。

2. **硬件与 AI 解耦**  
   终端负责采集和执行，服务器负责理解和决策，系统层次清晰。

3. **易扩展**  
   后续增加营养推荐、视觉分析、设备控制、用户画像和知识库时，可以优先扩展服务器。

4. **易部署**  
   可以本地调试，也可以部署到云服务器，支持异地设备接入。

5. **易维护**  
   模型配置、提示词、工具函数和业务参数集中管理，便于比赛期间快速迭代。

6. **适合比赛展示**  
   服务器可以清晰体现项目的软件智能能力，使硬件作品不只是机械执行系统，而是具备感知、理解和反馈能力的智能系统。

## 7. 总结

在本项目中，小智服务器是连接硬件设备和 AI 能力的关键中枢。它通过 OTA 和 WebSocket 管理设备接入，通过 ASR、LLM 和 TTS 完成语音交互，通过业务模块实现饮食推荐、参数计算或多模态辅助功能。

可以用一句话概括：

```text
硬件负责感知和执行，终端负责连接和上传，服务器负责理解、决策和生成反馈。
```

对于比赛提案而言，小智服务器体现了项目的软件架构能力和智能化水平。它让设备从单纯的硬件控制系统，升级为能够理解用户需求、生成个性化建议并驱动实际设备执行的智能交互系统。
